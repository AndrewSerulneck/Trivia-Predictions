## 1. Phased plan

| Phase | Goal | Key work | Model | Effort |
|---|---|---|---|---|
| **1** | Fix stuck expired-code pairing screen (Finding 1) | In `app/tv/page.tsx`: on `status === "expired"`, either (a) auto-remint a fresh code after a short delay instead of dead-ending polling, or (b) keep manual recovery but fix `TvPairingDisplay.tsx:125`'s copy to say something honest ("Code expired — tap below for a new one") and make the recovery action obvious/primary on the expired screen itself, not just the small corner link. Prefer (a) — auto-remint is the better UX for an unattended TV. Re-verify the `/tv` flow end-to-end (fresh mint → let a code expire → confirm recovery) using the `verify` skill / Playwright, the same way Phase 8 verified the rest of `/tv`. | **Sonnet 5** (isolated, well-scoped UI/state-machine fix, no architecture decisions) | Low–Medium |
| **2** | Fix final-standings replay-key collision (Finding 2) | Give each Live Trivia game a stable, unique identifier that survives across polls of the *same* game but differs between *different* games — `${scheduleId}:${occurrenceDate}` from `LiveShowdownState` is the natural candidate (both already exist on the engine state). Thread it through: add a `gameId: string` field to `VenueScreenState`'s live-trivia branch in `lib/venueScreen.ts` (mirrors how Phase 9 added `correctAnswer`/`revealEndsAt` — same pattern, same file); use it in `VenueScreenClient.tsx`'s `getScreenTransitionKey` for the `"final"` branch instead of bare `roundNumber`; pass it as the `gameId` prop into `TvFinalStandings` from `LiveTriviaIntermissionScreen.tsx` (currently omitted, which is *why* it falls back to the collision-prone `champion?.id`). Update the type + tests (same fixtures touched in Phase 9: `tests/lib.venue-screen.test.ts`, `tests/venue-screen-live-components.test.ts`, `tests/venue-screen-reliability.test.ts`, `lib/venueScreenDebug.ts`). Add a regression test: two consecutive `selectVenueScreenState` calls for different `occurrenceDate`s but identical `roundNumber`/champion must produce different `gameId`s. Browser-verify the celebration replays across two mocked back-to-back finals (same technique used to verify Phase 10's reveal beat — mock the poll API). | **Opus 4.8** (touches the core `VenueScreenState` type shape shared across the whole screen; needs the same care Phase 9 required) | Medium |
| **3** | Stop double-querying `resolveContinuousConfig` (Finding 3) | `lib/categoryBlitz.ts`'s `driveContinuousCategoryBlitz` already resolves the venue's continuous config internally (`lib/categoryBlitz.ts:2196`) but doesn't return it. Add the resolved config to `ContinuousSessionResult` (check all current callers/consumers of that type first — it's used elsewhere, not just `lib/venueScreen.ts`, so this is a shared-contract change) and have `lib/venueScreen.ts:338`'s `getCategoryBlitzInput` read `continuous.config` instead of calling `resolveContinuousConfig(venueId)` again. Keep the scheduled-path behavior unchanged. Confirm via a quick manual check (or a log/counter) that continuous-mode venues now issue one `category_blitz_continuous_config` read per poll, not two. | **Opus 4.8** (widens a shared return type used by more than one caller — needs the cross-file caller audit before changing the contract, similar risk profile to Phase 11's shared-anchor work) | Medium |
| **4** | Wire `revealEndsAt` for real, or delete it (Finding 4) | Decide once: (a) **wire it in** — have `LiveTriviaRevealScreen.tsx` derive its countdown from `live.revealEndsAt` (parse ISO, diff against `nowMs`) instead of the hardcoded `REVEAL_HOLD_SECONDS = 15`, so the TV's hold length is genuinely server-driven (matches the original Phase 9 design intent and the field's own doc comment); or (b) **delete the field** — remove `revealEndsAt` from the type, `selectVenueScreenState`, and all the test fixtures, and keep `REVEAL_HOLD_SECONDS` as the sole source of truth, updating its comment to stop claiming a server-driven countdown. Update the two direct unit-test assertions on `revealEndsAt` if the consumption logic changes their expected shape. Browser-verify the reveal beat still counts down correctly (reuse the Phase 10 mocked-poll technique). | **Sonnet 5** (mechanical once the direction is picked — no new architecture, just consuming an existing field) | Low |
| **5** | Centralize podium accent colors (Finding 5) | `lib/venueScreenBrand.ts` already exports `getPodiumAccent(rank)` (line 141) for this exact purpose. Recommend (a). This phase is really "Andrew decides, then Claude writes 2–3 sentences into CLAUDE.md" — no component code needs to change. | **Sonnet 5** (once the decision is made, it's a one-file doc edit) — **the decision itself needs Andrew**, not a model | Low (trivial once decided) |

## 2. Suggested order

**1 → 2 → 3 → 4 → (5 & 6 together) → 7 → 8**, with 8 able to move anywhere (it's independent of all the others and is a decision, not a fix).

Rationale: real user-facing correctness bugs first (1, 2), then the efficiency bug (3) since it's a live cost accruing on every poll, then the correctness/consistency gap (4), then the two reuse cleanups which touch overlapping files so are cheapest done together (5, 6), then the isolated simplification (7), with the policy question (8) parked at the end since it blocks nothing.

## 3. Verification expectations (per phase)

Every phase should close with the same gates used throughout the original feature work:
- `npx tsc --noEmit` clean
- `npx eslint <changed files>` clean (0 new errors)
- Relevant `vitest` suites green — at minimum `tests/lib.venue-screen.test.ts`, `tests/venue-screen-live-components.test.ts`, `tests/venue-screen-category-blitz-components.test.ts`, `tests/venue-screen-reliability.test.ts`; run the full `npm run test` before considering the whole plan done
- `npm run build` succeeds
- Any phase touching visual output (2, 4, 5, 6) should get a quick headless-Chromium screenshot pass (the same debug-mode-broadening + mocked-poll technique used in Phases 8–10 of the original plan) before/after, to catch regressions — revert any temporary debug scaffolding afterward, same discipline as before.
