# SMS After Subscribe Plan

## Goal

After a venue owner clicks **Subscribe — $100/mo** and completes Stripe Checkout, Hightop should send the post-subscription setup instructions by SMS instead of relying on the current welcome email.

Assumption to confirm: "the code" means the TV/display setup flow or TV pairing code currently promoted by the welcome email. If the desired SMS is a different code, define that before implementation.

## Current State

- Stripe Checkout is created by `app/api/owner/billing/checkout/route.ts`.
- Stripe confirms subscription activation through `app/api/webhooks/stripe/route.ts`.
- On `checkout.session.completed`, the webhook calls `maybeSendWelcomeEmail()`.
- Welcome email content lives in `lib/email/welcomeEmail.ts`.
- Owners currently have `name` and `email`, but no phone column. The registration UI at `app/owner/register/page.tsx` does not collect a phone number.
- There is no SMS provider integration yet.

## Recommended Provider

Use Twilio or another transactional SMS provider. Twilio is the default recommendation because it has good US SMS support, delivery status webhooks, and mature Node SDK/docs.

Required env vars will likely be:

- `TWILIO_ACCOUNT_SID`
- `TWILIO_AUTH_TOKEN`
- `TWILIO_MESSAGING_SERVICE_SID` or `TWILIO_FROM_NUMBER`
- Optional: `SMS_ENABLED=true`

## Phase 1 — Product Decision And Compliance

Owner: user

Intelligence required from agent: Low to Medium

Decisions:

- Confirm what should be texted:
  - TV pairing code
  - TV setup URL
  - owner dashboard URL
  - short welcome/setup message
- Decide whether SMS replaces email or supplements email.
- Decide whether phone is required before checkout or can be added later.
- Confirm consent language. US SMS requires clear opt-in language, even for transactional texts.
- Confirm business SMS registration needs, if applicable, such as A2P 10DLC for Twilio.

Recommended decision:

- Require phone during owner registration.
- Send a transactional SMS after successful subscription with a short setup URL, not a long marketing message.
- Keep email as a fallback until SMS delivery is verified in production.

## Phase 2 — Schema And Owner Profile

Owner: agent

Intelligence required from agent: Medium

Work:

- Add a new Supabase migration for `venue_owners`:
  - `phone_e164 text`
  - `sms_opted_in_at timestamptz`
  - optional `sms_opt_out_at timestamptz`
  - optional `welcome_sms_sent_at timestamptz`
- Do not edit historical migrations.
- Update owner registration API to accept `phone`.
- Validate and normalize phone to E.164 server-side.
- Update owner registration UI to collect phone.
- Add concise consent text near the phone field.

Potential follow-up:

- Add an owner profile/settings page so existing owners can add or update phone numbers.

## Phase 3 — SMS Provider Integration

Owner: user for provider account/env; agent for code

Intelligence required from agent: Medium

User work:

- Create/configure SMS provider account.
- Purchase or assign sending number/messaging service.
- Complete required compliance/brand registration if using Twilio A2P 10DLC.
- Add provider env vars to `.env.local` and Vercel Production/Preview.

Agent work:

- Install provider SDK if needed.
- Create `lib/sms/*` integration similar to existing `lib/email/*`.
- Make SMS sending no-op safely when env vars are missing.
- Add structured logging for success/failure without logging full phone numbers.

## Phase 4 — Message Content And Link/Code Generation

Owner: agent, with copy approval from user

Intelligence required from agent: Medium

Work:

- Create editable SMS copy file, for example `lib/sms/welcomeSms.ts`.
- Keep message short enough for SMS.
- Include:
  - venue name
  - setup action
  - short URL to owner dashboard/display setup
- If sending a TV pairing code specifically, generate or fetch the code at the correct time.

Open design question:

- TV pairing codes appear to be generated by the TV pairing flow, not simply available after payment. If the post-subscribe SMS must include a pairing code, implementation may need a new server-side code minting path or a link that opens the existing pairing flow.

Recommended implementation:

- Text a secure setup URL first:
  `Hightop Challenge is live for {venue}. Set up your TV: {url}`
- Avoid pre-generating TV pairing codes until the TV device is actually present.

## Phase 5 — Stripe Webhook Wiring

Owner: agent

Intelligence required from agent: High

Work:

- Update `app/api/webhooks/stripe/route.ts`.
- On `checkout.session.completed`:
  - upsert subscription as today
  - load owner and venue
  - if phone + opt-in exist, send welcome SMS
  - keep email fallback for missing phone or SMS failure
- Add idempotency guard:
  - use `welcome_sms_sent_at` or a dedicated notification log table
  - never send duplicate SMS on webhook retries
- Ensure SMS failure never fails the Stripe webhook after subscription sync succeeds.

## Phase 6 — Tests

Owner: agent

Intelligence required from agent: Medium to High

Work:

- Unit-test phone normalization and validation.
- Unit-test SMS copy generation.
- Unit-test webhook behavior:
  - sends SMS once on checkout completion
  - does not resend on duplicate webhook
  - falls back/no-ops safely when phone is missing
  - does not fail Stripe webhook on provider failure
- Keep existing Stripe status/invoice tests green.

## Phase 7 — Deployment And Verification

Owner: agent and user

Intelligence required from agent: High

User work:

- Confirm SMS provider production account is ready.
- Provide a real phone number for a controlled end-to-end test.
- Confirm the received SMS copy is acceptable.

Agent work:

- Apply Supabase migration.
- Add Vercel env vars.
- Deploy.
- Run a controlled live Stripe checkout.
- Verify:
  - `billing_subscriptions` active
  - SMS received once
  - email fallback behavior if enabled
  - Stripe webhook returns 200
  - provider delivery status is successful

## Phase 8 — Cutover

Owner: user and agent

Intelligence required from agent: Medium

Options:

- Keep email + SMS for the first few real venues.
- Once SMS delivery is reliable, make SMS primary.
- Leave email as backup for missing phone or SMS failures.

Recommended cutover:

- Do not fully remove the welcome email initially.
- Send SMS when available and keep email as a fallback/support artifact.
